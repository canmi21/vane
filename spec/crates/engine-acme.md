# vane-engine: ACME

Source: [`crates/engine/src/acme/`](../../crates/engine/src/acme/).

Automatic certificate issuance and renewal via the `ManagedCertPopulator`. Implements RFC 8555 (ACME core) plus RFC 9773 (ARI) over [`instant-acme`](https://crates.io/crates/instant-acme).

Listener TLS plumbing (cert resolver, populator trait, OCSP, rotation, session tickets) is in [`engine-tls.md`](engine-tls.md). This file covers only the ACME-specific parts.

Gated behind the `acme` Cargo feature. The Cloudflare DNS-01 provider is gated behind the additional `cloudflare` feature.

## Architecture

`ManagedCertRegistry` is daemon-scoped. A single registry lives for the daemon's lifetime and owns:

- ACME accounts keyed by `directory_url`.
- `instant-acme` clients.
- The pending-challenges table (consulted by the HTTP-01 responder and during the DNS-01 dance).
- The renewal scheduler.
- An `Arc<dyn AcmeStore>` for persistence.

`ManagedCertPopulator` (which implements `CertPopulator`) is FlowGraph-scoped — a view over the registry. At construction it tells the registry which SNIs the new graph wants managed; at `refresh()` it pulls cached cert state from the registry into a new `CertStore` for `ArcSwap` installation.

This separation isolates ACME state lifetime (daemon-scoped, survives reload) from cert delivery (FlowGraph-scoped, rebuilt per reload). Reload churn does not cause new order requests; rate-limit ceilings are not exposed to operator config-cycling.

```rust
pub struct ManagedCertRegistry {
    store:     Arc<dyn AcmeStore>,
    accounts:  DashMap<DirectoryUrlHash, Arc<AcmeAccount>>,
    pending:   DashMap<(Sni, ChallengeToken), PendingChallenge>,
    certs:     DashMap<Sni, Arc<RegisteredCert>>,
    schedule:  Arc<RenewalScheduler>,
}
```

Source: `crates/engine/src/acme/registry.rs`.

## `AcmeStore`

Persistence abstraction. Default impl: `FsAcmeStore` (filesystem). Other backends (object store, secrets manager) implement the same trait. Source: `crates/engine/src/acme/store.rs`, `crates/engine/src/acme/fs_store.rs`.

```rust
#[async_trait]
pub trait AcmeStore: Send + Sync {
    async fn load_account(&self, directory_url: &str) -> Result<Option<AcmeAccount>>;
    async fn save_account(&self, directory_url: &str, account: &AcmeAccount) -> Result<()>;

    async fn load_cert(&self, sni: &str) -> Result<Option<StoredCert>>;
    async fn save_cert(&self, sni: &str, cert: &StoredCert) -> Result<()>;
    async fn list_cert_snis(&self) -> Result<Vec<String>>;

    /// Acquire an exclusive advisory lock for `scope`. The returned guard
    /// releases on drop. Different `scope` strings are independent; the
    /// same `scope` serialises across both async tasks and OS processes.
    async fn lock(&self, scope: &str) -> Result<Box<dyn LockGuard>>;
}
```

The default fs impl uses `flock(2)` on a `.lock` file beside the target, plus a per-process `tokio::sync::Mutex` keyed by scope to close the same-process flock gap (multiple FDs in one process can hold the same flock concurrently on Linux).

### Storage layout

```
/var/lib/vaned/acme/
  accounts/
    <directory_url_hash>/
      account.json        # mode 0600
      .lock
  certs/
    <sni>/
      cert.pem            # leaf + intermediate chain
      key.pem             # mode 0600
      meta.json           # not_after, ari_replacement_id, last_renew_at
      .lock
```

`accounts/` is `0700`; private keys are `0600`; other files `0644`. `<directory_url_hash>` is `sha256(directory_url)[..16]` (hex). `<sni>` is the SNI lowercased with `*` replaced by `_wild_` for filesystem safety.

## Configuration schema

Rule-side config under `tls.managed`:

```jsonc
{
	"tls": {
		"sni": "api.example.com",
		"managed": {
			"directory_url": "https://acme-v02.api.letsencrypt.org/directory",
			"contact": ["mailto:ops@example.com"],
			"agree_tos": true,
			"challenge": "http-01",
			"key_type": "ecdsa-p256",
			"renew_before": "30d",
			"san": ["api.example.com", "www.api.example.com"]
		}
	}
}
```

Every field is required. JSON is generated by the CLI / TUI, not hand-written. Verbosity is free; absence is a compile error rather than an implicit default. Exceptions: `account_key_path` (BYO mode opt-in) and `dns_provider` (only when `challenge == "dns-01"`).

| Field              | Type                           | Required               |
| ------------------ | ------------------------------ | ---------------------- |
| `directory_url`    | string                         | yes                    |
| `contact`          | list\<string\>                 | yes                    |
| `agree_tos`        | bool (must be `true`)          | yes                    |
| `challenge`        | `"http-01"` \| `"dns-01"`      | yes                    |
| `dns_provider`     | object (provider-specific)     | iff `challenge=dns-01` |
| `account_key_path` | string                         | no (BYO)               |
| `key_type`         | `"ecdsa-p256"` \| `"rsa-2048"` | yes                    |
| `renew_before`     | duration                       | yes                    |
| `san`              | list\<string\>                 | yes                    |

Compile-time checks:

- `agree_tos != true` → error.
- `challenge == "dns-01"` without `dns_provider` → error.
- Wildcard label in `san` (`*.example.com`) but `challenge != "dns-01"` → error.
- `san` does not contain `tls.sni` → error.
- HTTP-01 challenge declared but no plaintext `:80` listener exists → warn (auto-bind attempted at runtime).

## Account key strategy

- Default: auto-create on first use of a `directory_url`. The generated key is persisted via `AcmeStore::save_account` and reused on subsequent boots.
- BYO via `account_key_path`: load PEM at boot; do not persist via store. Useful for migration and CI.
- An account is keyed by `directory_url` — vaned with two distinct directory URLs holds two accounts. Rare but supported.
- ToS acceptance is recorded in `AcmeAccount.agreed_tos_at` at registration. CA-side ToS-version bumps requiring re-acceptance surface as a registration error; the operator updates `agree_tos` (still `true`) and reloads to re-accept — treated as an active reaffirmation.

## SAN and wildcard

Each `tls.managed` block produces one cert covering all SANs in `san`. Two rules declaring identical `san[]` issue two distinct certs — multi-rule deduplication is the operator's responsibility.

Wildcard SANs require `dns-01`. ACME protocol constraint, not a vane policy.

Cert CN is `san[0]`; subsequent SANs are SAN-list only.

## Challenge: HTTP-01

ACME validators issue `GET http://<domain>/.well-known/acme-challenge/<token>` over plaintext H1.1 to the IP `<domain>` resolves to. `vaned` serves these via one of two mechanisms, picked at compile based on whether the operator's config has a plaintext `:80` listener.

### Case 1 — operator has a plaintext :80 listener

Compiler injects a high-priority synthetic route into every plaintext `:80` listener. Strict static predicate; non-ACME traffic unaffected.

```
listener.port      == 80
listener.kind       in { http, auto-plaintext }
http.version       == "HTTP/1.1"
http.method        == "GET"
http.uri.path     starts-with "/.well-known/acme-challenge/"
```

Fetch: `AcmeChallengeFetch { registry: Arc<ManagedCertRegistry> }`. Source: `crates/engine/src/fetch/acme_challenge.rs`. Priority above all operator rules on this listener.

Handler:

1. Extract the token from the path tail.
2. Look up `(Host header, token)` in `ManagedCertRegistry::pending`. Both checks required — a misrouted CA validator query could otherwise leak our key authorisation to a different virtual host.
3. Found → 200 OK with `Content-Type: application/octet-stream` and the key-authorisation body.
4. Not found → 404. The `/.well-known/acme-challenge/` namespace is reserved by vane whenever ACME is in use; falling through to operator rules would surface ACME plumbing as ordinary 404s.

The injected route is visible in `vane compile <DIR>` as `[acme-injected]` and in flow logs at runtime. Because of its priority, a `redirect_https` rule on the same listener does not swallow ACME validation traffic.

### Case 2 — no plaintext :80 listener

`vaned` attempts to bind a synthetic plaintext listener on `:80` (dual-stack). The synthetic listener serves only the ACME challenge route; anything else returns 404.

- Successful bind: `WARN` log "`acme: auto-bound :80 plaintext listener for HTTP-01 challenges; configure an explicit :80 listener to suppress this notice`".
- Bind failure (port in use, EACCES on a privileged port without `CAP_NET_BIND_SERVICE`): `ERROR` log with the OS error. Affected ACME issuances will fail HTTP-01; the registry surfaces the failure in `get_certs` (status `failed`, `last_error` populated). vaned does not abort boot — other functionality continues.

### Edge cases

- `:80` listener exists but is TLS-only (HTTPS on port 80, unusual): treated as "no plaintext :80"; auto-bind attempted and fails (port collision); ACME issuance fails until reconfigured. Compile-time warning emitted.
- Multiple plaintext `:80` listeners on different bind addresses (`0.0.0.0:80` and `192.168.1.5:80`): injected route lands in every one, since the CA validator's path through the network is non-deterministic.
- Plaintext `:80` listener with operator rules whose match would overlap `/.well-known/acme-challenge/`: injected node has higher priority, vane's responder wins. `compile <DIR>` annotates the operator's overlapping rule as `[shadowed-by-acme]`.

## Challenge: DNS-01

For domains where `:80` is unreachable (private networks, CGNAT, IPv6-only) or for wildcard certs.

### `DnsProvider` trait

Source: `crates/engine/src/acme/dns/`.

```rust
#[async_trait]
pub trait DnsProvider: Send + Sync + 'static {
    async fn set_txt(&self, fqdn: &str, value: &str) -> Result<TxtRecordHandle, DnsError>;
    async fn wait_propagated(&self, fqdn: &str, value: &str, timeout: Duration) -> Result<(), DnsError>;
    async fn delete_txt(&self, handle: TxtRecordHandle) -> Result<(), DnsError>;
}
```

Each impl defines its own `Config: serde::Deserialize` matching the rule-side `dns_provider` JSON.

`wait_propagated` should observe propagation via two channels and return when either fires:

1. Provider authoritative status — provider API's "TXT applied to authoritative servers" signal (e.g. Cloudflare's `success` flag).
2. Public resolver query — `8.8.8.8`, `1.1.1.1` for the TXT.

The CA validator queries DNS from its own resolvers; passing both is a high-confidence proxy. Default timeout 60 seconds; configurable per `dns_provider`.

### Cloudflare provider

Gated behind the `cloudflare` Cargo feature. Source: `crates/engine/src/acme/dns/cloudflare.rs`.

```jsonc
"dns_provider": {
  "kind":          "cloudflare",
  "api_token_env": "CF_API_TOKEN",   // env var holding the token
  "zone_id":       "abc123..."        // optional; auto-detected if omitted
}
```

Tokens are read from environment variables, never from JSON config — JSON is reloadable, the env var is set at startup. Matches the `.env`-vs-config split in [`core.md` § _Config layers_](core.md#config-layers).

## Challenge: TLS-ALPN-01 — not implemented

Vane does not implement TLS-ALPN-01. Its only unique scenario (`:80` unreachable AND DNS provider unavailable) is rare enough that the protocol-level disruption cost (special-case ALPN dispatch on `:443`, listener-side cert-resolver branching) is not worth supporting. Operators in that constraint use DNS-01.

## Renewal triggers

Three independent triggers may cause a renewal attempt. Source: `crates/engine/src/acme/scheduler.rs`.

### Periodic timer (primary)

Scheduler ticks every 5 minutes (matches `CertPopulator::refresh` cadence). For every registered cert:

- `now + renew_before >= not_after` → renew.
- ARI window set and `now ∈ window` → renew.
- Otherwise skip.

### ARI (RFC 9773)

If a cert came from a CA supporting ARI, its metadata includes a `renewalInfo` URL. The scheduler queries this URL after each successful issuance and records the CA-suggested `suggestedWindow.start..end`. Renewal triggers when wall-clock falls inside the window, regardless of `renew_before`.

ARI lets the CA spread renewal load and signal forced rotation (e.g. CA-side incident requiring early rotation). Honouring ARI is recommended by Let's Encrypt and is the 2025+ industry default.

Source: `crates/engine/src/acme/ari.rs`.

### `force_renew` mgmt verb

Operators can trigger immediate renewal:

```
vane cert renew api.example.com
```

Bypasses the periodic timer and ARI window; queues for immediate renewal. Useful for key-compromise rotation. See [`mgmt.md` § _Verbs_](mgmt.md#verbs).

## Rate limits and failure handling

Let's Encrypt limits (representative; other CAs vary):

- 50 certificates per registered domain per week.
- 5 duplicate certificates (identical SAN set) per week.
- 300 new orders per account per 3 hours.
- 10 account registrations per IP per 3 hours.

`ManagedCertRegistry` does not pre-empt these — caching last-known limits client-side is fragile. On a CA error of class `urn:ietf:params:acme:error:rateLimited`:

1. Records `last_error`, `next_attempt_at = now + backoff`, status = `limited`.
2. Backoff: exponential, base 30 minutes, factor 2, capped at 24 hours; resets to base on first success.
3. Logs `WARN` per backoff transition; subsequent retries during the same window are silent.

Other failure classes (network timeout, DNS provider error, validation timeout): same backoff, status = `failed`.

`get_certs` exposes per-cert state for operator visibility.

- Real Cloudflare testing is `#[ignore]`'d by default (requires a real zone and API token); CI runs on-demand via opt-in flag.
