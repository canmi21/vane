# Layer 7: HTTP Any-Bridge

## The invariants

**HTTP version any-combo.** Any client-side HTTP version can proxy to any upstream HTTP version. All nine combinations supported:

```
              upstream →  H1    H2    H3
client ↓
H1                        ok    ok    ok
H2                        ok    ok    ok
H3                        ok    ok    ok
```

**TLS is orthogonal.** On both client and upstream sides, HTTP or HTTPS are independent choices:

```
              upstream →  HTTP   HTTPS
client ↓
HTTP                      ok     ok
HTTPS                     ok     ok
```

**WebSocket is H1.1 on both sides** (WS over H2/H3 is permanently out of scope). WS/WSS combines same way:

```
              upstream →  ws     wss
client ↓
ws                        ok     ok
wss                       ok     ok
```

These three matrices are independent axes. Any combination across them is supported: e.g., a `wss://` client bridged to a `ws://` upstream, or an HTTPS/H3 client bridged to an HTTP/H1 upstream.

**WebSocket interaction with HTTP-version matrix**: WebSocket rules only match on L7 paths whose **client-side** HTTP version is H1.1. A listener that advertises `h2` or `h3` ALPN will never route a WebSocket upgrade — browsers (and every other spec-compliant H2/H3 client) do not emit HTTP Upgrade on H2/H3 connections, and RFC 8441 "WebSocket over H2" is permanently out of scope here. `vane lint` does **not** reject such rules at compile; they simply sit in the graph, unreachable. Author rules with WS predicates only on listeners explicitly configured for H1.1 or `Auto` (where ALPN may still negotiate H1.1).

## How the invariants hold

A consequence of type choice plus Fetch variant completeness:

- **Client-side ingress** — hyper (H1/H2) or the H3 path produces `http::Request<Body>`.
- **L7Request middleware** — operates on `http::Request<Body>`, agnostic to source version.
- **Fetch** — `Fetch::HttpProxy` consumes `http::Request<Body>` and forwards via:
  - `hyper_util::client::legacy::Client<_, Body>` for H1/H2 upstream (ALPN-negotiated, pooled).
  - `h3::client` for H3 upstream.
- **Response** flows back as `http::Response<Body>`.
- **L7Response middleware** — operates on `http::Response<Body>`.
- **Terminator `WriteHttpResponse`** — serializes the response using the client-side encoder.

Body is `http_body::Body<Data = Bytes>`. Frames flow from the client decoder directly into the upstream encoder (or from upstream decoder to client encoder on the return path) without `vaned` buffering.

Version translation (H1 `Host:` ↔ H2 `:authority` pseudo-header, chunked ↔ DATA frames) is owned by the corresponding client/server library. Vane does not touch pseudo-headers; hyper and h3 do.

### Cross-transport bridging is transport-free at the `http::Request` boundary

Any combination of client-side transport (TCP for H1/H2, QUIC-over-UDP for H3) with upstream-side transport is supported — not because the two sides coordinate, but because they **never meet at the transport layer**. The client-side socket is owned by the listener task; the upstream-side socket is owned by `TcpPool` or `QuicPool` (see below). They only meet at the `http::Request<Body>` value, which contains no transport reference. So an H1-over-TCP client request can be bridged to an H3-over-UDP upstream (or any of the 9 version cells) without a dedicated "translation" path: one end produces `http::Request<Body>`, the other consumes it.

### `HttpProxyFetch` commits to streaming response bodies

For the "native stream when possible" posture (see later section) to be architecturally guaranteed — not merely permitted — `HttpProxyFetch` must return the upstream's response body in its original streaming variant: `Body::Http12(hyper::body::Incoming)` for H1/H2 upstreams, `Body::Http3(H3Body)` for H3 upstreams. It never collects the upstream response body into `Body::Static` as a defensive measure. The only exception: an upstream response with zero frames may arrive as `Body::Empty` (semantically equivalent to a zero-length `Body::Static`). `HttpSynthesizeFetch` always produces `Body::Static` by construction — that is the point of synthesis.

### Upstream-H3 send path

For any client-side version sending to an H3 upstream, `HttpProxyFetch` reads the request `Body` via `http_body::Body::poll_frame`. For `Frame::data(Bytes)` frames, the `Bytes` is handed directly to `h3::client::RequestStream::send_data(Bytes)` — h3 accepts `impl Buf`, so this is ownership transfer, not copy. `Frame::trailers(HeaderMap)` becomes `h3::client::RequestStream::send_trailers(HeaderMap)`. The reverse direction (H3 upstream response → any-version client) drives `H3Body` (see `03-types.md`) through `poll_frame`; the client-side encoder consumes it frame by frame.

### TLS orthogonality — how it works

**Client-side HTTPS** means the listener terminates TLS. SNI-based cert resolution runs at L4→L7 upgrade (see `08-tls.md`). After upgrade, the L7 pipeline sees plaintext `http::Request<Body>` — TLS is invisible above this point.

**Upstream-side HTTPS** means `HttpUpstream::Tcp { tls: Some(UpstreamTls {...}) }`. The connector wraps its TCP stream in `rustls::ClientConfig` before handing to hyper. Middleware and Fetch see plaintext.

Both together: both wrappings apply, no special-case code. The middleware layer is TLS-agnostic.

### WS/WSS orthogonality

Identical pattern. Listener TLS termination covers the `wss://` client side; `WsUpstream::Tcp { tls: Some(...) }` covers the `wss://` upstream side. The WebSocket handshake and post-upgrade byte tunnel are plaintext from Vane's perspective either way.

## Upstream connection pool

Where the any-bridge stops being a type contract and starts being an operational promise. The pool's structure and lifetime rules drive every upstream interaction.

### Architecture: TCP / QUIC separation

Two daemon-level pool systems, one per transport family:

```
vaned
├─ TcpPool   (singleton; hyper-util Client with ConnectorStack)
│   └─ per-destination connection cache, keyed by TcpFingerprint
└─ QuicPool  (singleton; per-destination h3 client manager)
    └─ per-destination QUIC connection, keyed by QuicFingerprint
```

TCP pooling is delegated entirely to `hyper_util::client::legacy::Client`, which keys its internal pool by authority. We do not run per-connection accounting at the TCP layer.

QUIC is per-destination: each upstream gets one `h3::client::Connection` multiplexing many streams. The destination → connection map lives in `vaned` because `h3` ships no pool abstraction.

### Pool fingerprint

Two `Fetch::HttpProxy` instances share a pool slot iff their fingerprints match:

```
TcpFingerprint  = (addr, version_slot, tls_hash)
QuicFingerprint = (addr, tls_hash)

version_slot = Auto | Http1 | Http2 | Http3
tls_hash     = hash(root_CA_source, client_cert, crls, verify_mode, alpn_protocols)
```

`verify_mode` and `crls` participate in the fingerprint because Full vs Skip verification and different CRL sets are distinct trust boundaries — pools cannot be shared across them.

Two rules proxying to `example.com:443` with the same root CA share the pool. Any fingerprint parameter differing — especially the TLS hash — allocates a separate slot, preventing TLS configuration cross-contamination.

### Lifetime: daemon-level

Pool structures exist for the daemon's lifetime. FlowGraph reload does not touch them.

Connection-level lifetime is governed by **idle timeout** — a connection sitting idle beyond `idle_timeout` (default `30s`) is closed by the pool. When a FlowGraph reload removes the last rule referencing some upstream, nothing special happens: its connections simply idle out over the next 30 seconds.

Reference-counted pools (last dropped Fetch closes the pool) are deliberately not used — the complexity exceeds the benefit; the cost is transient idle connections for a few seconds after reload.

`max_lifetime` (forced rotation regardless of activity) is out of scope for MVP. Post-MVP adds a management verb `pool.drain <fingerprint>` for forced close — useful after cert rotation.

### Exhaustion defaults (per upstream, configurable)

| Parameter                 | Default | Meaning                                               |
| ------------------------- | ------- | ----------------------------------------------------- |
| `max_idle_per_host`       | `32`    | Cap on idle connections per authority                 |
| `max_concurrent_per_host` | `100`   | Cap on concurrent in-flight connections per authority |
| `connect_timeout`         | `5s`    | Upper bound on establishing a new upstream connection |
| `idle_timeout`            | `30s`   | Idle connection close threshold                       |

When all connections are busy and `max_concurrent_per_host` is hit, new requests wait internally. A wait exceeding `connect_timeout` returns `Upstream` ErrorKind → `503 Service Unavailable`.

### DNS resolver: `hickory-resolver`

Upstream address resolution uses [`hickory-resolver`](https://github.com/hickory-dns/hickory-dns) (renamed `trust-dns`) instead of hyper-util's default `GaiResolver`.

|                         | `hickory-resolver` (chosen)      | `GaiResolver` (rejected)                         |
| ----------------------- | -------------------------------- | ------------------------------------------------ |
| Implementation          | pure Rust DNS client             | system `getaddrinfo` via blocking thread         |
| Async                   | native                           | blocking, wrapped in `spawn_blocking`            |
| TTL caching             | yes                              | none                                             |
| System config honored   | `/etc/resolv.conf`, `/etc/hosts` | full NSS path (nsswitch, systemd-resolved, mDNS) |
| musl static-link clean? | yes                              | no (glibc NSS baggage across distros)            |
| Custom DNS servers      | supported per-upstream           | requires OS-level config                         |

Default behavior reads the OS's DNS settings. A per-upstream override may point resolution at a specific DNS server — useful for split-horizon deployments where internal upstreams resolve through a private DNS and external ones use the system default.

The musl compatibility is a real operational win: `vaned` statically linked against musl (common for container images) does not carry glibc's NSS baggage and behaves consistently across Alpine, Debian, and RHEL-family distros.

### Stream fan-out: H2/H3 client → H1 upstream

When the client multiplexes N concurrent streams on one connection and the upstream is H1 (no multiplexing), the pool opens multiple upstream TCP connections up to `max_concurrent_per_host`. Each stream checks out a connection, sends the request, awaits the response, returns the connection. When saturated, streams wait bounded by `connect_timeout`.

### Stream fan-in: H1 client → H2/H3 upstream

Each H1 request becomes a single H2/H3 stream on the shared upstream connection. The upstream connection is reused across many client requests; no change on the client-facing side.

### Error classification (retry-relevant)

The retry decision consumes this table:

| ErrorKind                         | Retryable | Reasoning                                                                          |
| --------------------------------- | :-------: | ---------------------------------------------------------------------------------- |
| Connection refused                |    yes    | Transient                                                                          |
| Connection reset — mid-request    |    no     | Request may have partially committed; replay unsafe                                |
| Connection reset — on idle pickup |    yes    | Race: grabbed an idle conn the peer had just closed                                |
| Connect timeout                   |    yes    | Transient                                                                          |
| H2 `RST_STREAM REFUSED_STREAM`    |    yes    | Upstream explicitly signalling "try again"                                         |
| H2/H3 `GOAWAY` (stream > last_id) |    yes    | Upstream said "didn't process this stream" — replay safe regardless of idempotency |
| TLS handshake fail                |    no     | Configuration error, not transient                                                 |
| DNS resolution fail               |    yes    | Commonly transient; default one retry                                              |
| Response 4xx / 5xx                |    no     | Complete response; retry would duplicate the request                               |
| Malformed upstream HTTP           |    no     | Protocol error, not recoverable                                                    |

`Response 4xx/5xx` is explicitly **not** retry-eligible — those are valid HTTP responses, just unfavorable ones; retrying them duplicates requests without basis.

### H2 / H3 `GOAWAY` handling

Upstream connections may announce `GOAWAY` with a `last_stream_id`. For any in-flight request whose stream id is greater, the upstream explicitly did **not** process the request — it is safe to replay regardless of method idempotency.

Mapping:

- `hyper-util::Client` (H2) surfaces this as an error variant we translate to `Error::upstream(UpstreamReason::Gone)` at the Fetch boundary.
- The `h3::client` path does the same for H3.
- `is_retryable()` returns `true` for `Gone`; the Fetch's retry loop handles it transparently if retry is enabled. Otherwise it propagates to `Node::Middleware.on_error` (none on Fetch node, but the default fail-safe tombstone applies — 502 to the client).
- No separate vane-side bookkeeping: the hyper / h3 client owns the `last_stream_id` tracking; we only see the resulting per-stream error.

Established streams with id ≤ `last_stream_id` are allowed to complete normally. The upstream connection is retired from the pool once its last in-flight stream finishes; new streams fingerprint the same authority and go through a fresh connect.

## Body streaming across versions

H1 chunked, H2 DATA frames, and H3 DATA frames unify under `http_body::Body::poll_frame`. The `Body` enum delegates to the underlying protocol body. `Fetch::HttpProxy` reads `Frame<Bytes>` from the client body via `http_body::Body::poll_frame` and hands each `Bytes` to the upstream encoder **without vane-layer copy or buffering**. QUIC reassembly (packet coalescing, stream-offset merging), H2 flow-control window accounting, and H1 chunk decoding are owned by `quinn` / `hyper` / `h3` internally; vane neither interposes on nor claims to eliminate those costs. The response path follows the same rule. See `03-types.md` § _What vane's "no copy" covers (and what it does not)_ for the precise scope.

## Body streaming: native when possible, buffered when required

Vane streams bodies end-to-end **iff neither LazyBuffer track fires on the path** (see `02-flow.md` § _LazyBuffer_). Support is conditioned on the combination of endpoints:

| Combo                     | Streaming posture                                                                                                                                                          |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| H2 ↔ H2, H3 ↔ H3, H2 ↔ H3 | **Native bidirectional streaming.** Multiplexed transports support partial-frame handoff in both directions concurrently.                                                  |
| H1.1 ↔ {H1.1, H2, H3}     | Half-duplex streaming per direction; client-side reuse requires H1.1 keep-alive.                                                                                           |
| HTTP/1.0 client ↔ any     | Client side is one-shot (see below). Upstream side is unaffected.                                                                                                          |
| Any ↔ CGI                 | Upstream cannot stream the request side (RFC 3875 requires stdin EOF before the child produces output). Response side streams as the child writes stdout. See `15-cgi.md`. |

**LazyBuffer firing on either side degrades that side to buffered.** The other side remains streaming. The executor uses the same body-sink and body-source regardless of which posture each side is in; the only runtime difference is whether the in-flight body is `Body::Static` (buffered) or `Body::Http12 / Http3 / Stream` (streaming). Vane's promise is "native stream when possible" — not "always stream".

### HTTP/1.0 client-side degradation

A client speaking HTTP/1.0 (detected by request line version) cannot participate in keep-alive reliably. `Connection: keep-alive` on H1.0 is an optional extension rarely honored by real 1.0 clients, and the default is connection-per-request. Vane therefore treats an H1.0 client connection as **one-shot**: after `WriteHttpResponse` completes, the client-side TCP connection is closed. The "H1 ↔ H2 keep-alive" guarantee in the table above applies to H1.1 only. Upstream-side connection reuse (H1.1 keep-alive on the pool, H2/H3 multiplexing) is unaffected; the pool is keyed by upstream authority, not by client connection.

## WebSocket

`Fetch::WebSocketUpgrade`. Both sides H1.1. See `05-terminator.md` for the handshake extraction pattern (extracting `OnUpgrade` before destructuring).

After upgrade, the path ends in `Terminator::ByteTunnel` — Vane duplex-copies bytes between the two upgraded streams. No frame parsing; application payload semantics are not our concern.

## Upstream types

All three upstream transports reach `Fetch::HttpProxy` via the `HttpUpstream` variant:

- **TCP** — ordinary reverse proxy. H1/H2/H3 via `version` field.
- **Unix socket** — a custom hyper connector dials the Unix path. H1/H2 only (UDP-transport protocols over Unix are not meaningful).
- **CGI** — spawns a subprocess per request. Request body → stdin; stdout → response (headers then body, RFC 3875 parsing). A CGI-specific connector bridges `http::Request` into process marshaling.

CGI is the outlier. Every other upstream is connection-pooled and keep-alive-capable; CGI is fork-exec per request. Users choose CGI deliberately (legacy integration).

## Timeouts (proposal)

Each `Fetch::HttpProxy` carries:

- `connect_timeout` — upstream connection establishment. Default `5s`.
- `read_timeout` — between bytes when reading upstream response. Default `30s`.
- `total_timeout` — overall request budget. Default `60s`.
- `max_body_size` — upper bound on buffered body when LazyBuffer compiled this path to buffered. Default `8 MiB`.

Defaults are conservative. Overrides live in the rule's Fetch config.

## Retry policy

Retry lives inside Fetch. Configuration knobs in `05-terminator.md`.

Architectural consequences:

- Retry requires request-body replay; the body must be `Body::Static` or `Body::Empty`. Rules opting into retry automatically mark their path for eager buffering — users do not configure this separately.
- Retry scope is one `HttpUpstream`. Cross-upstream failover is expressed via multiple rules (each with its own Fetch) plus fallback predicates in the FlowGraph, not via a retry mechanism.
- Retry attempts count against pool occupancy. A rule configured with `max_attempts: 5` holding a busy pool can block other rules sharing the same upstream; operators size pools accordingly.
