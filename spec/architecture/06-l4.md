# Layer 4

## Listeners

```rust
pub struct Listener {
    pub transport: Transport,        // Tcp | Udp
    pub addr:      std::net::SocketAddr,
    pub kind:      ListenerKind,
}

pub enum ListenerKind {
    Raw,   // all traffic is L4; FlowGraph at this port is L4-only
    Http,  // all traffic is HTTP; TLS termination if TLS detected
    Auto,  // peek and decide per-connection
}
```

Listeners bind at boot. Listener-set changes (add / remove) happen via config reload diff — see [`01-topology.md`](01-topology.md) for the full lifecycle. A dedicated management-API verb for standalone listener add/remove outside of reload is post-MVP.

### Listener kind derivation

`ListenerKind` is **derived from each listener's FlowGraph entry subgraph** during compile — users do **not** configure it directly. There is no `listener_kind` field in the rule schema; the graph shape is the single source of truth. This matches the rest of `vane`'s "graph-form determines behavior" pattern (predicate inspection level, `flow_graph::needs_peek`, the phase state machine).

Derivation rule, applied per listener address against the union of the entry subgraphs of every rule that listens on that address:

| Reachable terminator set                                      | Derived kind |
| ------------------------------------------------------------- | ------------ |
| Only L4 fetches (`L4Forward`, etc.)                           | `Raw`        |
| Only L7 fetches (every L4→L7 path crosses an `Upgrade` node)  | `Http`       |
| Both L4 and L7 fetches reachable from the same listener entry | `Auto`       |

A graph with **no** terminators (only middleware / unreachable rule body) is a compile error elsewhere and never reaches the derivation step. The derivation produces a single `ListenerKind` per `SocketAddr`; dual-stack `listen: ":443"` (which expands to two entry keys sharing one `NodeId`) yields the same kind for both v4 and v6 listeners.

Per-preset derivation outcome is documented in [`14-presets.md`](14-presets.md). Mixed listeners (one preset expanding to L4 + another to L7 on the same port) produce `Auto`.

#### UDP listener semantics

The `ListenerKind` enum is shared between TCP and UDP listeners; the variant names are interpreted per transport:

| Variant | TCP                                                                     | UDP                                                                                                                                                            |
| ------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Raw`   | All traffic is L4 byte forwarding                                       | All traffic is L4 datagram forwarding (no QUIC awareness)                                                                                                      |
| `Http`  | All traffic is HTTP/1.1 or HTTP/2 (TLS-terminated when a cert is bound) | All traffic is HTTP/3 — QUIC handshake terminated by `quinn`, H3 streams over it                                                                               |
| `Auto`  | Peek discriminates between TLS, H1, H2, raw bytes                       | Peek discriminates between QUIC initial (→ H3 termination), QUIC initial with `tls.sni` predicate (→ L4Forward by SNI), and other UDP (→ L4Forward by 4-tuple) |

Operators declare the wire transport via the `udp:` prefix on a listen entry (see [`09-config.md`](09-config.md) § _Transport prefix_). They do not write `protocol: "h3"` or any equivalent rule-level flag. Given a `udp:` listener, the listener kind is then derived from the graph: rules using HTTP-level fetches yield `Http` (= H3-over-QUIC); rules using only `L4Forward` yield `Raw`; a mixed listener yields `Auto`. Whether multi-packet peek is required for `Auto` is decided by graph form — see § _Multi-packet peek_ below.

## Protocol detection

The peek phase reads up to a small prefix (default 8 KiB, or until a detector is satisfied) and runs built-in detectors:

- **TLS ClientHello** — first byte `0x16`, valid record envelope. SNI and ALPN extracted without decryption.
- **HTTP/1 request line** — ASCII method (`GET`, `POST`, etc.) + SP + path + SP + `HTTP/1.`.
- **HTTP/2 preface** — exact bytes `PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`.
- **QUIC initial packet** (UDP only) — long-header form with a recognized version.
- **DNS query** (UDP only) — valid DNS header layout.
- **Unknown** — opaque bytes, routed to the L4 subgraph.

Detection results populate a `PeekResult` struct inserted into `ConnContext.user` before any middleware runs. Predicates matching on `peek` (raw bytes) or structured fields (`tls.sni`, `tls.alpn`) read from the appropriate derived location — see below.

```rust
pub struct PeekResult {
    pub buffer:   bytes::Bytes,              // raw peeked bytes — read by `peek` predicates
    pub detected: Option<DetectedProtocol>,  // first detector that claimed the bytes
    pub tls:      Option<TlsClientHello>,    // parsed ClientHello if the bytes look like TLS
}

pub enum DetectedProtocol {
    TlsClientHello,
    Http1,
    Http2Preface,
    QuicInitial,
    Dns,
    Unknown,
}

pub struct TlsClientHello {
    pub sni:   Option<String>,   // extracted SNI extension
    pub alpn:  Vec<Vec<u8>>,     // ALPN protocol IDs offered by the client
}
```

When `PeekResult.tls` is populated, the peek step **writes `sni` into `ConnContext.tls`** (with `alpn`, `version`, and `peer_cert` left `None` until the full handshake completes). This gives predicates a single canonical path — `tls.sni` always reads from `ConnContext.tls.sni`. `tls.alpn` and `tls.version` are post-handshake values (the negotiated protocol and TLS version respectively); pre-handshake code that needs the client's ALPN offer list reads `PeekResult.tls.alpn` directly. The raw `peek` buffer remains accessible for byte-level predicates on non-TLS protocols.

### Dispatch decision table

The listener combines the derived `ListenerKind`, the `PeekResult.detected` value, and whether a `rustls::ServerConfig` is bound to this address (`listener_tls`) to choose the per-connection dispatch path. `listener_tls.is_some()` is itself a graph-derived property — it is `Some` iff at least one rule with a `tls` block lists this address. The full table:

| Kind   | `detected`               | `listener_tls` | Dispatch                                                                                                                          |
| ------ | ------------------------ | -------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `Raw`  | any                      | any            | Hand the `PeekedStream` (or raw stream when `needs_peek=false`) to the L4 subgraph; **no handshake**. SNI passthrough lives here. |
| `Http` | `TlsClientHello`         | `Some`         | TLS handshake (`run_tls`), then L7.                                                                                               |
| `Http` | `TlsClientHello`         | `None`         | **Reject** — graph claims L7 but cannot decrypt.                                                                                  |
| `Http` | `Http1` / `Http2Preface` | any            | **Reject** — `Http`-derived listeners do not accept cleartext on the wire (mixed-posture is `Auto`'s job).                        |
| `Http` | `Unknown`                | any            | **Reject** — opaque bytes have no L4 fallback in `Http`.                                                                          |
| `Auto` | `TlsClientHello`         | `Some`         | TLS handshake, then L7.                                                                                                           |
| `Auto` | `TlsClientHello`         | `None`         | Hand to the L4 subgraph if one is reachable from this entry; otherwise **reject**. (SNI passthrough.)                             |
| `Auto` | `Http1`                  | any            | Cleartext H1 — `hyper::server::conn::http1::Builder` directly on the peeked stream.                                               |
| `Auto` | `Http2Preface`           | any            | Cleartext H2 (h2c) — `hyper::server::conn::http2::Builder` directly on the peeked stream.                                         |
| `Auto` | `Unknown`                | any            | Hand to the L4 subgraph if one is reachable; otherwise **reject**.                                                                |

`detected = None` (the prelude returned indeterminate after exhausting `MAX_PEEK_BYTES`) is treated as `Unknown` for the purposes of this table.

Listeners whose FlowGraph has no reachable `L4Peek` middleware skip the peek prelude entirely (`needs_peek=false`); for those the dispatch is `(kind, listener_tls)` only:

- `Http+Some` → handshake then L7.
- `Http+None` → L4 subgraph fallthrough. The graph reaches `Node::Upgrade`, which drives the cleartext stream through `hyper::server::conn::http1::Builder`. This is the **cleartext HTTP reverse-proxy** path (e.g. `listen: [":80"]` with an `http_proxy` terminator and no `tls` block) — a legal, common configuration where the operator has decided the wire is plaintext on purpose. Non-HTTP traffic delivered to this path fails closed via hyper's parse error rather than via an explicit pre-protocol reject; the connection still drops without leaking application bytes, so the operator-visible behavior is equivalent to `reject` for the misconfigured / hostile cases. The peeked path (`needs_peek=true`) keeps the strict reject for `Http+H1/H2/Unknown` because there the operator opted into per-connection inspection and explicit rejection is what they asked for.
- `Raw+any` → L4 subgraph on raw stream.

`Auto` listeners by construction have at least one `L4Peek` reachable (the protocol detection node), so `needs_peek=false` is impossible for them.

**Rejection** at this stage closes the TCP connection without writing application bytes. There is no synthetic 4xx — the peer protocol may not even be HTTP. A `tracing::debug!` event records the reason; the L1 floor still counts the connection in its per-IP / global tallies for backoff purposes. `vane`'s answer to "I want a 4xx for unknown protocols on `:443`" is to add an L4 forward rule pointing at a synth-response upstream — i.e., express the policy as a graph, not a built-in.

## `udp_dispatch`

`vane` owns the **physical** UDP socket and demultiplexes packets to one of several handlers. Each UDP listener holds a dispatch table; every datagram pulled from the physical socket is looked up in that table before any FlowGraph work happens:

```
UDP packet arrives on listener's physical socket
  ↓
dispatch table lookup
  ├─ hit (hot path)
  │   ├─ entry = L4Forward session  → push datagram to forwarder's inbound channel
  │   └─ entry = QUIC virtual socket → push datagram to that virtual socket's inbound queue
  │                                    (quinn drives the connection from there)
  ├─ pending-peek hit (cold path, in formation — see § Multi-packet peek)
  │   └─ append datagram to pending buffer; try ClientHello extraction; on completion,
  │      enter FlowGraph carrying all buffered datagrams
  └─ miss (cold path)
      ├─ packet has a QUIC long header AND listener is `Http` UDP?
      │                                          → fan into the listener's per-listener virtual socket
      │                                            (drives `quinn::Endpoint`); skip FlowGraph entry
      ├─ listener requires multi-packet peek?    → create pending-peek entry, accumulate
      └─ otherwise                               → build L4Conn::Udp(UdpAssoc { socket, peer,
                                                     first_packets: vec![datagram] }) and
                                                     enter FlowGraph immediately
        ↓ matched terminator
        └─ L4Forward     → fetch dials upstream, sends every datagram in `first_packets`
                           in order, spawns forwarder, registers handle in dispatch table
```

Hot-path packets never touch the FlowGraph. Cold-path packets enter exactly once per session — the registration step closes the loop, so every subsequent datagram from the same peer (L4Forward) goes hot. QUIC packets on `Http` UDP listeners short-circuit FlowGraph entry entirely: `quinn::Endpoint` handles the connection lifecycle (accept, handshake, stream demultiplexing, migration, close) internally, with `vane` only routing inbound datagrams into the endpoint's virtual socket and never inspecting QUIC connection state.

`udp_dispatch` is **implementation-level, not user-visible**. Users do not configure session demux; they configure listeners and rules, and the demux falls out of which terminators the rules elect.

### UDP socket multiplexing: physical and virtual

`vane` does not give any handler exclusive ownership of the physical UDP socket. Instead it interposes a multiplexer:

| Layer             | What owns it                   | Lifetime                                               |
| ----------------- | ------------------------------ | ------------------------------------------------------ |
| Physical socket   | UDP listener                   | Listener lifetime (boot → reload diff)                 |
| Dispatch table    | UDP listener                   | Listener lifetime; entries come and go                 |
| L4Forward session | Forwarder task                 | Until idle timeout, listener cancellation, or peer EOF |
| Virtual socket    | `quinn::Endpoint` per listener | Listener lifetime (Http UDP listeners only)            |

`quinn::Endpoint` expects to own a `quinn::AsyncUdpSocket`-shaped object. Rather than handing it the physical socket — which would prevent any other UDP traffic on the same port from reaching `vane` — the listener interposes a single **virtual socket** wrapper per `Http` UDP listener that filters and forwards:

- recv: the listener's recv loop classifies each inbound datagram by packet form. QUIC long-header packets (and short-header packets belonging to QUIC connections the endpoint already accepted) fan into the virtual socket's inbound queue; non-QUIC datagrams take other paths (L4Forward / pending-peek / unknown-drop).
- send: the virtual socket forwards `quinn`'s outbound datagrams to the physical socket's `send_to` (no copy beyond what the trait demands).

`vane` does **not** track QUIC connection identifiers in the dispatch table. CID demultiplexing — including the server-issued CID rotation, multi-CID lifecycle, NEW_CONNECTION_ID frame handling, and migration across peer-address changes — happens entirely inside `quinn::Endpoint`'s internal state. From `vane`'s perspective the endpoint is opaque: the recv loop pours QUIC packets in, the endpoint's internal connection registry routes them to the right `quinn::Connection`. This is what makes `vane` a transparent proxy in front of `quinn` — `quinn` does not know `vane` is interposed; `vane` does not need to peek into `quinn`'s connection state.

This shape makes one physical UDP port able to serve any combination of L4 byte forwards, QUIC connections, DNS forwarding, and unknown-protocol drops simultaneously — the same `mux port over UDP` discipline TCP+TLS+H1+H2c gets on the listener side.

Dispatch keys split by handler kind:

```rust
enum DispatchKey {
    Peer(SocketAddr),         // L4Forward sessions: identity is (listener_addr, peer)
    PendingPeek(SocketAddr),  // Cold-path session in formation; see § Multi-packet peek
}
```

QUIC connections do not appear as dispatch-table entries. The `Http` UDP listener's recv loop short-circuits QUIC packets straight to the per-listener `quinn::Endpoint`'s virtual socket; `quinn::Endpoint` then performs its own CID-keyed demultiplexing internally. For QUIC SNI passthrough scenarios (where `vane` routes by SNI without terminating QUIC — see § _Multi-packet peek_), the resulting passthrough session is held as an `L4Forward` entry keyed by `Peer(SocketAddr)`, not by CID; `quinn` is not involved in those sessions at all.

Both `L4Forward.idle_timeout` and `quinn::Endpoint`'s own idle-timeout knob remain authoritative for their respective handlers — the dispatch table itself does not own session timing (see _UDP idle timeout is single-authority_ below).

#### Why one virtual socket per listener (and not per connection)

An earlier draft of this design proposed one `VirtualUdpSocket` per QUIC connection, with `vane` maintaining a CID-keyed dispatch entry per connection and subscribing to `quinn-proto`'s endpoint-event stream to register new server-issued CIDs. That design is **not implementable** against the public `quinn` / `quinn-proto` API: `quinn_proto::EndpointEvent` is an opaque wrapper around a `pub(crate)` enum, exposing only `is_drained()` externally. The CID-rotation events that the per-connection model required are not surfaced.

The per-listener model imposes no operational cost: `quinn::Endpoint` already maintains its own connection registry and routes CIDs internally; the only thing the per-connection design would have added was a redundant `vane`-side CID table that duplicated `quinn`'s internal table. Mixed L4Forward + H3 on the same listener works fine under the per-listener model because the recv loop classifies by packet form (QUIC long-header vs other UDP) before fanning into the virtual socket. Migration across NAT rebind works because `quinn` handles it internally; `vane` does not need to observe the migration.

### Multi-packet peek

QUIC's TLS ClientHello can span multiple Initial packets — common when the ClientHello carries large key shares (post-quantum hybrid groups push it well past one MTU). A single-datagram `first_packet` model is sufficient for L4 forwards keyed on peer 4-tuple, and for H3 termination (where `quinn` reassembles internally and `vane` does not need to read the ClientHello). It is **not** sufficient for the QUIC SNI passthrough case, where `vane` must extract the SNI from a possibly-fragmented ClientHello before routing to the right L4 upstream.

The cold-path therefore supports a third state, **pending-peek**, sitting between miss and active dispatch: datagrams accumulate, a built-in detector tries to extract the SNI as each datagram arrives, and the FlowGraph entry is delayed until the SNI is known (or a bound is exceeded).

#### When pending-peek activates

Compile-time derivation, similar to `needs_peek` for TCP listeners:

| Listener graph shape (UDP)                                                            | Pending-peek required? |
| ------------------------------------------------------------------------------------- | :--------------------: |
| L4Forward routes only, no `tls.sni` predicate                                         |           no           |
| Any rule on the listener uses `tls.sni` predicate **and** has an L4Forward terminator |          yes           |
| H3 termination only (every L7 path on this listener crosses an `Upgrade` to H3)       |           no           |
| Mixed: L4Forward-with-`tls.sni` plus H3 termination                                   |          yes           |

On listeners where pending-peek is not required, cold-path entry happens immediately on the first datagram (existing model, `first_packets` carries one datagram). On listeners where it is required, every cold-path datagram first goes through the pending-peek state machine.

#### State machine

```
Cold-path datagram arrives, no active dispatch entry
  ↓
Recognize as QUIC long-header Initial?
  ├─ no  → fall through to immediate cold-path entry (the listener's other rules
  │        still work; e.g. DNS-over-UDP or non-QUIC L4Forward continue normally)
  └─ yes → look up pending-peek entry for this peer 4-tuple
            ├─ entry exists:
            │     append datagram to buffer
            │     try ClientHello extraction over accumulated buffer
            │     ├─ complete: enter FlowGraph with `first_packets = buffer`,
            │     │             `ctx.tls.sni = parsed`; remove pending entry
            │     └─ incomplete: leave entry; await more datagrams
            └─ no entry:
                  create entry { buffer: [datagram], started_at: now }
                  try extraction (single datagram is the common-case complete ClientHello)
                  ├─ complete: enter FlowGraph immediately, remove entry
                  └─ incomplete: keep entry; await more
```

#### Bounds (fixed; not configurable)

| Bound                                        | Value | Rationale                                                                         |
| -------------------------------------------- | ----- | --------------------------------------------------------------------------------- |
| Max bytes per pending session                | 16 KB | Real ClientHellos rarely exceed 8 KB even with PQ key shares; 16 KB has headroom. |
| Max datagrams per pending session            | 8     | Worst-observed real fragmentation is 2–3 packets; 8 is generous.                  |
| Pending session lifetime                     | 1 s   | Covers cross-continent RTT with margin; longer suggests attack or bug.            |
| Max concurrent pending sessions per listener | 1024  | Resource cap against floods. New sessions arriving past the cap drop silently.    |

Exceeding any per-session bound (bytes, datagrams, lifetime) drops the pending entry. The peer's natural retry (QUIC clients retry handshake on timeout) gives a clean second chance.

#### Indexing key

Pending entries are keyed by **peer 4-tuple** (peer address + listener address). QUIC peer migration does not happen during the Initial exchange — clients only migrate after the handshake completes and they have negotiated server-issued CIDs. NAT collisions during the same millisecond on the same public mapping are vanishingly rare; the cost of a collision is one retried handshake.

The client's Source Connection ID is **not** used as the indexing key during pending-peek: the first datagram has not yet been parsed when the lookup happens, so the SCID is not available at lookup time. Once parsed, the SCID is captured in `ctx.tls` as part of the FlowGraph entry but does not become a dispatch index.

#### ClientHello extraction

Initial packet payloads are encrypted with a key derived from the connection's initial Destination Connection ID (RFC 9001 §5.2), so any party with the DCID can decrypt — no secret material is required. `vane` delegates the parser to the workspace's [`clienthello`](../../crates/lib/clienthello) crate (under `crates/lib/`, MIT, independently publishable) instead of reaching into `quinn-proto` / `rustls` internals. Rationale:

- `quinn_proto::initial_keys` and the wider Initial-keys derivation surface are `pub(crate)` — not callable from outside `quinn-proto`. The earlier "reuse `quinn_proto::initial_keys`" plan was not implementable against the public API.
- Hand-rolling the four labels' worth of HKDF-Expand-Label per RFC 9001 §5.1 / §5.2 is ~50 LoC of standards-fixed code: salt and labels are pinned to the RFC and will not drift across `quinn` releases.
- Avoiding `rustls::server::Acceptor` keeps the parser independent of rustls's evolving session API and lets the crate stay deployable without a TLS termination stack on the consumer side.

Internally `clienthello` does:

- QUIC long-header parsing per RFC 9000 §17.2 (Initial-only).
- HKDF-Expand-Label key derivation per RFC 9001 §5.1 / §5.2 against the v1 initial salt.
- AES-128-ECB header protection per RFC 9001 §5.4 + AES-128-GCM payload AEAD per RFC 9001 §5.3.
- QUIC frame walking (CRYPTO / PADDING / ACK / PING; any other frame type in an Initial packet is a protocol violation and drops the pending session).
- Offset-keyed CRYPTO byte-stream reassembly across multiple Initial datagrams.
- TLS ClientHello SNI extension extraction per RFC 6066 §3 (the ClientHello arrives as raw handshake bytes — QUIC carries no TLS record header, so the parser starts from the HandshakeMessage).

Crypto backend is RustCrypto's `aes-gcm` / `aes` / `ctr` / `hkdf` / `sha2` / `subtle` — pure-Rust, no C deps, friendly to downstream consumers. The crate carries no `vane-*` dependency and is shaped to be adoptable by any project needing the same primitive.

Extraction returns `Some(sni)` when a complete ClientHello has been observed and SNI parsed; `None` otherwise (more datagrams needed). Errors during decryption or parsing (malformed packet, version mismatch) drop the pending session — `vane` does not attempt recovery.

QUIC v2 (RFC 9369) is mechanical to add (different initial salt + TLS 1.3 cipher suite) and is not implemented in 0.1.0 of `clienthello`.

#### Replay to handler

Multi-packet peek's purpose is **routing decision**, not QUIC termination — the matched terminator for a pending-peek session is `L4Forward` (typically a SNI-passthrough rule that forwards bytes to a chosen upstream by SNI). H3-termination listeners do not require multi-packet peek (the listener-derivation rules ensure their entry path doesn't activate it), since `quinn::Endpoint` reassembles fragmented Initials internally for connections it terminates.

Once the FlowGraph routes a pending session, every buffered datagram is delivered in original arrival order to the chosen `L4Forward` handler: the forwarder's first action after dial-up is to send each `first_packets[i]` to the upstream, in order, before subscribing to the inbound mpsc for hot-path datagrams.

Dispatch-table registration happens before the first packet is replayed, so any new datagrams from the same peer 4-tuple hit the active entry on the hot path. There is no window where a packet falls between the buffered batch and the registered hot path.

## `l4_forward`

The simple case: accept a connection, forward bytes to an upstream. TCP uses `tokio::io::copy_bidirectional`; UDP uses a session-scoped packet forwarder keyed by 5-tuple.

L4 middleware may run before the forward terminates. Example: an SNI peek middleware reads the TLS ClientHello, populates `ctx.tls.sni`, then the Terminator picks an upstream based on SNI — without ever decrypting. This is the SNI passthrough pattern common to load balancers fronting multiple TLS tenants.

### UDP idle timeout is single-authority

For UDP, the session-scoped forwarder needs an idle timeout to reclaim 5-tuple entries whose peer has silently stopped. **That timeout is `L4Forward.idle_timeout` — the single, authoritative source.** `udp_dispatch`'s session demultiplexer does **not** maintain its own competing timeout: sessions it creates for `L4Forward` paths are timed by the forwarder itself; sessions it creates for QUIC termination have their lifecycle governed by the QUIC stack (`quinn`) and its own idle-timeout knob. No `min(a, b)` races, no ambiguity about which counter ticks when.

Users tune a single `idle_timeout` on the rule; operators reading `get_connections` see one TTL per connection, which matches the configured value.

## Zero-copy at L4

`copy_bidirectional` can use `splice(2)` on Linux, moving bytes between kernel buffers without copying into userspace. When middleware peeks (e.g., SNI extraction), the peeked prefix is copied into a small buffer; bytes past the peek remain zero-copy.

## L4 → L7 upgrade

When a FlowGraph path crosses from L4 into L7 (e.g., "TCP accept → peek ClientHello → TLS terminate → parse HTTP"), the upgrade is ordered:

1. **Decrypt** (if TLS): rustls consumes the TLS record layer, exposing a plaintext `AsyncRead + AsyncWrite`. `ctx.tls` is populated.
2. **Negotiate**: ALPN result determines HTTP version. `ctx.http_version` is populated.
3. **Parse**: hyper (for H1/H2) or the H3 path (for H3) consumes bytes, producing `Request` objects.
4. **Dispatch**: each `Request` is handed to the L7 subgraph of the FlowGraph.

The upgrade is one-way. Once a connection is L7, it is L7 for its lifetime. Upgrade failure (malformed TLS, ALPN mismatch) drops the connection; there is no L7-to-L4 fallback.

## Listener lifecycle

Listener add / remove happens at two points: **boot** and **reload**. Both go through the same mechanism — the daemon diffs the next compiled listener set against the current one, spawning new listeners with bind-retry and cancelling removed ones with soft-drain. Full mechanics (bind retry, drain semantics, IPv4/IPv6, privileges) in [`01-topology.md`](01-topology.md).

A dedicated management-API verb for standalone listener add / remove outside of reload is post-MVP. The existing task model needs no refactor to support it — each listener already runs as an independent tokio task with a cancellation token.
