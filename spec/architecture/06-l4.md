# Layer 4

## Listeners

```rust
pub struct Listener {
    pub transport: Transport,        // Tcp | Udp
    pub addr:      std::net::SocketAddr,
    pub kind:      ListenerKind,
}

pub enum ListenerKind {
    Raw,   // all traffic is L4; FlowGraph at this port is L4-only
    Http,  // all traffic is HTTP; TLS termination if TLS detected
    Auto,  // peek and decide per-connection
}
```

Listeners bind at boot. Listener-set changes (add / remove) happen via config reload diff — see [`01-topology.md`](01-topology.md) for the full lifecycle. A dedicated management-API verb for standalone listener add/remove outside of reload is post-MVP.

## Protocol detection (Auto listeners)

The peek phase reads up to a small prefix (default 8 KiB, or until a detector is satisfied) and runs built-in detectors:

- **TLS ClientHello** — first byte `0x16`, valid record envelope. SNI and ALPN extracted without decryption.
- **HTTP/1 request line** — ASCII method (`GET`, `POST`, etc.) + SP + path + SP + `HTTP/1.`.
- **HTTP/2 preface** — exact bytes `PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`.
- **QUIC initial packet** (UDP only) — long-header form with a recognized version.
- **DNS query** (UDP only) — valid DNS header layout.
- **Unknown** — opaque bytes, routed to the L4 subgraph.

Detection results populate a `PeekResult` struct inserted into `ConnContext.user` before any middleware runs. Predicates matching on `peek` (raw bytes) or structured fields (`tls.sni`, `tls.alpn`) read from the appropriate derived location — see below.

```rust
pub struct PeekResult {
    pub buffer:   bytes::Bytes,              // raw peeked bytes — read by `peek` predicates
    pub detected: Option<DetectedProtocol>,  // first detector that claimed the bytes
    pub tls:      Option<TlsClientHello>,    // parsed ClientHello if the bytes look like TLS
}

pub enum DetectedProtocol {
    TlsClientHello,
    Http1,
    Http2Preface,
    QuicInitial,
    Dns,
    Unknown,
}

pub struct TlsClientHello {
    pub sni:        Option<String>,          // extracted SNI extension
    pub alpn:       Vec<Vec<u8>>,            // offered ALPN protocol IDs
    pub versions:   Vec<u16>,                // supported_versions extension
}
```

When `PeekResult.tls` is populated, the peek step **also writes `sni` and `alpn` into `ConnContext.tls`** (with `version` and `peer_cert` left `None` until the full handshake completes, if any). This gives predicates a single canonical path — `tls.sni` always reads from `ConnContext.tls.sni` — instead of having to probe both pre- and post-handshake sources. The raw `peek` buffer remains accessible for byte-level predicates on non-TLS protocols.

## `udp_dispatch`

UDP on `Auto` or `Http` listeners passes through a session demultiplexer before entering the FlowGraph:

```
UDP packet arrives
  ↓
session table lookup
  ├─ hit   (part of an established QUIC association)
  │         → hot path: route directly to session handler (quinn / h3)
  └─ miss  (new or non-QUIC)
          → cold path: construct L4Conn, enter FlowGraph
```

The session table is owned by whoever terminated the session (normally `quinn`). `udp_dispatch` is an implementation-level concern, not user-visible — users do not configure UDP session demux. It is how UDP listeners multiplex stateful and stateless traffic on the same socket.

Hot-path packets never touch the FlowGraph. Cold-path packets enter once (to establish whatever state they imply) and subsequent packets for the same session are hot-path.

## `l4_forward`

The simple case: accept a connection, forward bytes to an upstream. TCP uses `tokio::io::copy_bidirectional`; UDP uses a session-scoped packet forwarder keyed by 5-tuple.

L4 middleware may run before the forward terminates. Example: an SNI peek middleware reads the TLS ClientHello, populates `ctx.tls.sni`, then the Terminator picks an upstream based on SNI — without ever decrypting. This is the SNI passthrough pattern common to load balancers fronting multiple TLS tenants.

## Zero-copy at L4

`copy_bidirectional` can use `splice(2)` on Linux, moving bytes between kernel buffers without copying into userspace. When middleware peeks (e.g., SNI extraction), the peeked prefix is copied into a small buffer; bytes past the peek remain zero-copy.

## L4 → L7 upgrade

When a FlowGraph path crosses from L4 into L7 (e.g., "TCP accept → peek ClientHello → TLS terminate → parse HTTP"), the upgrade is ordered:

1. **Decrypt** (if TLS): rustls consumes the TLS record layer, exposing a plaintext `AsyncRead + AsyncWrite`. `ctx.tls` is populated.
2. **Negotiate**: ALPN result determines HTTP version. `ctx.http_version` is populated.
3. **Parse**: hyper (for H1/H2) or the H3 path (for H3) consumes bytes, producing `Request` objects.
4. **Dispatch**: each `Request` is handed to the L7 subgraph of the FlowGraph.

The upgrade is one-way. Once a connection is L7, it is L7 for its lifetime. Upgrade failure (malformed TLS, ALPN mismatch) drops the connection; there is no L7-to-L4 fallback.

## Listener lifecycle

Listener add / remove happens at two points: **boot** and **reload**. Both go through the same mechanism — the daemon diffs the next compiled listener set against the current one, spawning new listeners with bind-retry and cancelling removed ones with soft-drain. Full mechanics (bind retry, drain semantics, IPv4/IPv6, privileges) in [`01-topology.md`](01-topology.md).

A dedicated management-API verb for standalone listener add / remove outside of reload is post-MVP. The existing task model needs no refactor to support it — each listener already runs as an independent tokio task with a cancellation token.
