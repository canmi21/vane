# Layer 4

## Listeners

```rust
pub struct Listener {
    pub transport: Transport,        // Tcp | Udp
    pub addr:      std::net::SocketAddr,
    pub kind:      ListenerKind,
}

pub enum ListenerKind {
    Raw,   // all traffic is L4; FlowGraph at this port is L4-only
    Http,  // all traffic is HTTP; TLS termination if TLS detected
    Auto,  // peek and decide per-connection
}
```

Listeners bind at boot. MVP does not support hot add/remove.

## Protocol detection (Auto listeners)

The peek phase reads up to a small prefix (default 8 KiB, or until a detector is satisfied) and runs built-in detectors:

- **TLS ClientHello** â€” first byte `0x16`, valid record envelope. SNI and ALPN extracted without decryption.
- **HTTP/1 request line** â€” ASCII method (`GET`, `POST`, etc.) + SP + path + SP + `HTTP/1.`.
- **HTTP/2 preface** â€” exact bytes `PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`.
- **QUIC initial packet** (UDP only) â€” long-header form with a recognized version.
- **DNS query** (UDP only) â€” valid DNS header layout.
- **Unknown** â€” opaque bytes, routed to the L4 subgraph.

Detection results populate a `PeekResult` struct inserted into `ConnContext.user` before any middleware runs. Predicates can match against this.

The session table is owned by whoever terminated the session (normally `quinn`). `udp_dispatch` is an implementation-level concern, not user-visible â€” users do not configure UDP session demux. It is how UDP listeners multiplex stateful and stateless traffic on the same socket.

Hot-path packets never touch the FlowGraph. Cold-path packets enter once (to establish whatever state they imply) and subsequent packets for the same session are hot-path.

## `l4_forward`

The simple case: accept a connection, forward bytes to an upstream. TCP uses `tokio::io::copy_bidirectional`; UDP uses a session-scoped packet forwarder keyed by 5-tuple.

L4 middleware may run before the forward terminates. Example: an SNI peek middleware reads the TLS ClientHello, populates `ctx.tls.sni`, then the Terminator picks an upstream based on SNI â€” without ever decrypting. This is the SNI passthrough pattern common to load balancers fronting multiple TLS tenants.

## Zero-copy at L4

`copy_bidirectional` can use `splice(2)` on Linux, moving bytes between kernel buffers without copying into userspace. When middleware peeks (e.g., SNI extraction), the peeked prefix is copied into a small buffer; bytes past the peek remain zero-copy.

## L4 â†’ L7 upgrade

When a FlowGraph path crosses from L4 into L7 (e.g., "TCP accept â†’ peek ClientHello â†’ TLS terminate â†’ parse HTTP"), the upgrade is ordered:

1. **Decrypt** (if TLS): rustls consumes the TLS record layer, exposing a plaintext `AsyncRead + AsyncWrite`. `ctx.tls` is populated.
2. **Negotiate**: ALPN result determines HTTP version. `ctx.http_version` is populated.
3. **Parse**: hyper (for H1/H2) or the H3 path (for H3) consumes bytes, producing `Request` objects.
4. **Dispatch**: each `Request` is handed to the L7 subgraph of the FlowGraph.

The upgrade is one-way. Once a connection is L7, it is L7 for its lifetime. Upgrade failure (malformed TLS, ALPN mismatch) drops the connection; there is no L7-to-L4 fallback.

## Listener lifecycle

Listener add / remove happens at two points: **boot** and **reload**. Both go through the same mechanism â€” the daemon diffs the next compiled listener set against the current one, spawning new listeners with bind-retry and cancelling removed ones with soft-drain. Full mechanics (bind retry, drain semantics, IPv4/IPv6, privileges) in [`01-topology.md`](01-topology.md).

A dedicated management-API verb for standalone listener add / remove outside of reload is post-MVP. The existing task model needs no refactor to support it â€” each listener already runs as an independent tokio task with a cancellation token.
