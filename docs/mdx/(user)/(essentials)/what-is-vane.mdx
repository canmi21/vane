Vane operates as a **transparent execution layer** between clients and upstreams.

| Plane             | Responsibility                                                                                      |
| ----------------- | --------------------------------------------------------------------------------------------------- |
| **Data Plane**    | Accepts connections from Clients, executes user-defined logic, and optionally connects to Upstreams |
| **Control Plane** | Drives behavior through atomic configuration updates, applied without dropping connections          |

## The Execution Lifecycle

Vane does not follow a fixed `receive → route → forward` pipeline. Instead, when a connection arrives, it triggers a programmable sequence of events:

<Steps>
<Step>
**Accept**
A new connection `TCP`/`UDP` is established on a configured listener. At this moment, it is just a raw stream of bytes without meaning.
</Step>

<Step>
	**Context Initialization** To track this connection's journey, an execution **Context** is
	immediately initialized. This mutable container will hold all `state`, `metadata`, `variables`,
	and `buffers` as the connection progresses.
</Step>

<Step>
	**Flow Entry** The connection is then handed over to a **Flow** which is the user-defined decision
	graph specific to this port. From here on, the engine follows your instructions, not a hardcoded
	path.
</Step>

<Step>
	**Plugin Execution** Plugins execute sequentially against the Context. As each plugin runs, it
	inspects the state, potentially modifies it, and **decides** which branch of the graph the
	connection should traverse next.
</Step>

<Step>
**Outcome**
This loop of inspection and decision-making continues until a terminal action is reached, finalizing the connection's fate.
</Step>
</Steps>

## Routing is Optional

<Callout type="warn">
	It is critical to understand that **routing to an upstream is just one possible outcome** of a
	Flow.
</Callout>

Vane is not fundamentally a "router". It is an engine for executing logic. Depending on the path taken through your Flow, a connection might:

| Outcome                       | Description                                                                         |
| ----------------------------- | ----------------------------------------------------------------------------------- |
| **Be routed**                 | Connection forwarded to an upstream server after passing all checks                 |
| **Be rejected**               | Connection terminated immediately when a security policy triggers or by your desgin |
| **Upgrade its layer**         | Protocol upgraded (e.g., `TCP` → `TLS`) for deeper inspection before final decision |
| **Receive a direct response** | Vane constructs synthetic payload (e.g., `JSON`) without touching any backend       |

The decision to route or to respond directly is not a system default; it is a choice delegated entirely to your Flow definition.

## Terminology Boundaries

To effectively use Vane, you must distinguish between these core concepts.

<Accordions type="single">
<Accordion title="Flow: The Decision Graph">
The definition of the decision graph. A Flow is a tree of steps where each step executes a Plugin and branches based on the result.
</Accordion>

<Accordion title="Plugin: The Unit of Logic">
The atomic unit of logic. Plugins are divided into two roles:

- **Middleware**: Performs work (inspection, modification) and returns a branch to continue execution.
- **Terminator**: Performs a final action (proxy, respond, abort) that ends the current Flow branch.

</Accordion>

<Accordion title="Layer: The Level of Abstraction">
The level of abstraction at which the engine is currently operating:

| Layer                 | Description                                                   |
| --------------------- | ------------------------------------------------------------- |
| **Transport Layer**   | Operating on raw IP/Port pairs                                |
| **Carrier Layer**     | Operating on protocol headers without decryption              |
| **Application Layer** | Operating on fully parsed application messages (e.g., `HTTP`) |

</Accordion>

<Accordion title="Upgrade: Moving Between Layers">
	The mechanism for moving a connection from a lower layer to a higher layer. An upgrade preserves
	the physical connection but replaces the execution Context with one capable of understanding the
	new protocol.
</Accordion>

<Accordion title="Context: The Corss-Layer KV Store">
The mutable state carrier for a single connection. It persists across Plugin executions and accumulates metadata (e.g., `Client IP`, `TLS SNI`, `User ID`) as the Flow progresses.
</Accordion>
</Accordions>
