L4+ uses a specific flow management strategy called **Resolvers**. Unlike L4 Listeners which bind to ports, Resolvers bind to protocol names (e.g., `tls`, `http`, `quic`).

## Architecture & Lifecycle

<Steps>
<Step>

### Registry Structure

Resolvers are stored in a global, thread-safe registry powered by `ArcSwap`. This allows lock-free reads during high-concurrency traffic.

<Mermaid
	chart="
graph LR
    Reg[RESOLVER_REGISTRY] -->|Atomic Load| Map{DashMap}
    Map -- tls --> TLS_Flow[TLS Config]
    Map -- http --> HTTP_Flow[HTTP Config]
    Map -- quic --> QUIC_Flow[QUIC Config]
"
/>

</Step>
<Step>

### Atomic Hotswap

When a configuration file changes in `resolver/`, the Hotswap system triggers a reload.

<Mermaid
	chart="
graph LR
    Watcher[File Watcher] -->|Event| Hotswap[Hotswap Task]
    Hotswap -->|Scan| Disk[Load All Resolvers]
    Disk -->|Build| NewMap[New Registry Map]
    NewMap -->|Swap| Global[Update ArcSwap]
    Global -->|Drop Old| Cleanup[Free Old Configs]
"
/>

</Step>
<Step>

### Execution Context

When a flow starts, the L4+ engine initializes a specialized `TransportContext`.

<Mermaid
	chart="
graph LR
    Init[Initialize Context] --> KV[Set conn.layer = l4p]
    Init --> Payloads[Inject Payloads]
    Payloads -->|tls.clienthello| Context
    Payloads -->|http.headers| Context
    Context --> Exec[Execute Flow]
"
/>

</Step>
</Steps>
