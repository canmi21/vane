The Service Discovery module manages the registry of upstream nodes that Vane uses for load balancing and proxying. It transforms static configuration into a high-performance, flattened runtime state.

## Data Model & Hierarchy

The configuration (typically defined in `nodes.json` or `nodes.toml`) follows a multi-tiered hierarchy to allow flexible backend definitions.

- **`NodesConfig`**: The top-level container.
- **`Node`**: A logical grouping of backends (e.g., "web-cluster").
- **`IpConfig`**: A specific IP address associated with one or more ports.

### Pre-Processing & Flattening

To ensure the proxying path is as fast as possible, Vane does not traverse the nested node tree during request processing. Instead, it uses a **Flattening Strategy** during the loading phase.

<Mermaid
  chart='
graph LR
    subgraph Input [Hierarchical Config]
        NodeA["Node: web"] --> IP1["IP: 1.1.1.1"]
        IP1 --> P1["Port: 80"]
        IP1 --> P2["Port: 443"]
    end

    Input -->|Flattening| Output

    subgraph Output [Runtime Registry]
        PN1["web | 1.1.1.1:80"]
        PN2["web | 1.1.1.1:443"]
    end

'
/>

The `PreProcess` trait transforms the tree into a flat list of `ProcessedNode` entries:

```rust title="src/resources/service_discovery/model.rs"
pub struct ProcessedNode {
    pub node_name: String,
    pub address: String,
    pub port: u16,
    pub ip_type: IpType,
}
```

If a node has 2 IPs and each IP specifies 3 ports, the flattener generates 6 discrete `ProcessedNode` entries.

## Configuration Validation

Vane performs rigorous validation during the loading process:

1.  **Unique Names**: Every node must have a unique identifier.
2.  **Formatting**: Node names are restricted to lowercase letters, numbers, and hyphens.
3.  **Network Sanity**: IP addresses must be valid.

### Atomic Storage

The parsed and validated config is stored in `NODES_STATE`, a global `ArcSwap<NodesConfig>`.

<Mermaid
	chart="
graph LR
    Parsed[Parsed Config] --> Pre[Pre-Process: Flattening]
    Pre --> Validate{validate}
    Validate -- Ok --> Store[Store in NODES_STATE]
    Validate -- Err --> Log[Log Error / Keep Old]
"
/>

</Step>
<Step>

### Real-time Updates

The `hotswap.rs` module runs a dedicated watch loop to detect file changes and perform zero-downtime swaps.

<Mermaid
	chart="
graph LR
    Notify[Watcher Event] --> Scan[Re-scan File]
    Scan --> Diff{Content Changed?}
    Diff -- Yes --> Swap[Atomic Swap Pointer]
    Diff -- No --> Ignore[No-op]
"
/>

</Step>
</Steps>

<Callout type="info" title="Effective Changes Only">
	Vane's hotswap logic compares the actual content of the new configuration against the old one. If
	you save the file without making changes, the engine will skip the expensive re-calculation and
	swap.
</Callout>
