Vane operates on a strictly tiered architecture, processing connections at different levels of the OSI model depending on the required depth of inspection.

The engine distinguishes between **Three Layers**, each with its own execution context and capabilities.

## The Three Layers

### 1. Layer 4 (Transport)
**Source**: `src/layers/l4/`

The entry point for all connections. It operates on raw **TCP Streams** and **UDP Datagrams**.

- **Focus**: Routing based on IP, Port, and initial packet sniffing.
- **Context**: `TransportContext` (Connection metadata, peek buffer).
- **Actions**: Proxy (TCP/UDP), Deny, or **Upgrade** to a higher layer.
- **No Decryption**: It cannot see inside encrypted traffic (TLS/QUIC) beyond the initial handshake.

### 2. Layer 4+ (Carrier)
**Source**: `src/layers/l4p/`

The "Carrier" layer handles protocol inspection **without full termination**. It parses handshake headers to make intelligent routing decisions for encrypted protocols.

- **Focus**: TLS SNI, ALPN, QUIC Connection IDs.
- **Context**: `TransportContext` (Same as L4, but with protocol-specific metadata).
- **Actions**: Route based on SNI/Host, Proxy, or **Upgrade** to L7.
- **Partial Parsing**: It parses *just enough* to route (e.g., ClientHello), then tunnels or hands off.

### 3. Layer 7 (Application)
**Source**: `src/layers/l7/`

The Application layer provides full **HTTP Termination**. It parses the entire request, allowing for deep inspection and modification.

- **Focus**: HTTP Headers, Body, Path, Query, Methods.
- **Context**: `Container` (Full Request/Response envelope).
- **Actions**: Middleware transformation, Synthetic Responses, Upstream Fetching.
- **Full Termination**: The connection is terminated at Vane; a new connection is made to the upstream.

## Layer Transitions (Upgrading)

Connections move up the stack via an **Upgrade** mechanism. This is triggered by a `Terminator` plugin returning an upgrade signal.

<Steps>
<Step>
### L4 Ingress
A TCP connection arrives. The **L4 Flow** peeks at the first few bytes.
Plugin `detect.tls` sees a TLS ClientHello.
</Step>

<Step>
### Upgrade Signal
The L4 flow executes an **Upgrade Terminator** (e.g., `upgrade "tls"`).
The engine suspends L4 processing and wraps the stream in a TLS parser.
</Step>

<Step>
### L4+ Routing
The **L4+ Flow** (`tls.json`) parses the SNI (e.g., `example.com`).
It decides this domain requires WAF processing.
</Step>

<Step>
### Final Upgrade
The L4+ flow executes `upgrade "httpx"`.
The engine performs the TLS handshake, decrypts the traffic, and hands the plaintext stream to the **L7 Engine**.
</Step>
</Steps>
