The `KvStore` is a high-performance, per-connection storage space used to accumulate metadata as a connection traverses the different layers of the Vane engine. It serves as the primary source of truth for the [Template System](./templates) and plugin decision-making.

## Implementation

Technically, the `KvStore` is a type alias for `ahash::AHashMap<String, String>`.

- **AHashMap**: Vane uses `ahash` because it is significantly faster than the standard library's `HashMap` for small-to-medium string keys, which is critical for the hot path of plugin execution.
- **String Storage**: To maintain maximum compatibility and simplicity for templates and external plugins, all values are stored as strings.

## Lifecycle

The `KvStore` follows the strict lifecycle of the network connection it is attached to.

<Steps>
<Step>

### Creation Phase

A new store is initialized in the `ingress` layer immediately after a TCP or UDP connection is accepted.
At this stage, standard metadata is populated: `conn.uuid`, `conn.ip`, `conn.proto`, etc.

</Step>
<Step>

### Mutation & Processing

The `KvStore` is owned by the connection task (the flow).
As the connection traverses layers:

- **L4**: Adds routing decisions.
- **L4+**: Adds protocol headers (SNI, Host).
- **L7**: Middleware plugins inject results and headers.

Throughout this process, the [Template System](./templates) reads from the store to resolve dynamic configuration values.

</Step>
<Step>

### Destruction Phase

When the connection terminates (client disconnects or server closes), the task ends.
The `KvStore` is automatically dropped, releasing all memory back to the system.

</Step>
</Steps>

## Standard Connection Keys

Upon initialization, Vane pre-populates the store with essential metadata under the `conn` namespace:

| Key              | Description                                              | Example         |
| :--------------- | :------------------------------------------------------- | :-------------- |
| `conn.uuid`      | A unique UUIDv7 (ordered) identifier for the connection. | `018d1a...`     |
| `conn.ip`        | The source (client) IP address.                          | `192.168.1.100` |
| `conn.port`      | The source (client) port number.                         | `54321`         |
| `conn.proto`     | The transport protocol (`tcp` or `udp`).                 | `tcp`           |
| `conn.timestamp` | The Unix timestamp of when the connection was accepted.  | `1736340000`    |

## Key Scoping & Conventions

To prevent collisions between different subsystems and plugins, Vane follows a strict dot-notated naming convention:

- **`conn.*`**: Core transport-layer metadata.
- **`tls.*`**: Metadata extracted during L4+ TLS inspection (e.g., `tls.sni`).
- **`http.*`**: Metadata from HTTP protocol handlers (e.g., `http.host`).
- **`plugin.<path>.<name>.*`**: Keys added by specific plugins. The engine automatically handles this scoping to ensure that two instances of the same plugin in different branches do not overwrite each other's data.

<Callout type="info" title="Performance Tip">
	While the `KvStore` is fast, avoid storing large blobs of data (like full request bodies) directly
	as keys. Instead, use the store for metadata and rely on the execution context for payload access.
</Callout>
